## Core Workflow

Use this skill to solve OR, OM, computer-science optimization, and mathematical modeling problems from problem intake through final high-quality decision artifacts. The main goal is not to prove that code can run; it is to design, implement, and evaluate a policy, optimizer, or analytical method that is meaningfully useful for practitioners under the user's stated quality, runtime, and deployment constraints.

Treat runnable code as a necessary deployment property, not as the method-selection objective. For high-stakes sequential OR tasks, especially stochastic MDP-like decisions, a simple hand-built heuristic or random parameter search is only a primary method when the skill has first shown that higher-ceiling analytical, solver-based, dynamic programming, reinforcement learning, or hybrid methods are structurally unnecessary or infeasible under the user's constraints.

4. After the user confirms the formulation, prefer parallel exploration. If multi-agent or subagent tooling is available, launch three independent specialist agents in parallel:

6. After all branches complete, run reviewer synthesis using `references/reviewer.md`. The reviewer must compare branch scores, branch labels, candidate-method names, and narrative recommendations carefully; any "highest ranked" or "recommended" statement must be consistent with the comparison table unless the reviewer explicitly documents an override rationale. The reviewer must optimize for the best deployable problem-solving method under the confirmed formulation, not merely the simplest runnable output. For MDPs, stochastic dynamic programs, simulator-backed sequential decision problems, and complex nonconvex stochastic control tasks, serious learned methods such as PPO, actor-critic, MAPPO/RMAPPO, fitted value iteration, tree/regression policies, and other parametric policy/value approximators must be considered unless the user explicitly forbids black-box or semi-black-box models. For nonlinear production or clearing-function models, serious solver routes such as dynamic or rolling-horizon MILP/MINLP, piecewise-linear approximations, decomposition, and lower-bound/hindsight optimization must be considered unless model scale or licensing makes them infeasible. Method selection must include a performance-ceiling versus implementation-effort frontier and an acceptance-target plan: strong baselines, expected improvement criterion, tuning and model-size or solver-strength rationale, tuning budget, and what evidence would reject the proposed method. Method selection must also keep primary methods, fallbacks, bounds, and baselines conceptually separate. Baselines should normally be classical, literature-standard, current-practice, simple-rule, naive, or lower-bound / hindsight benchmarks that are materially simpler or methodologically different from the proposed primary model, but they must still be implemented as competent comparators: tune material heuristic parameters, solver limits, thresholds, or scenario counts within a fair budget, record the tuning rule, and avoid intentionally weak defaults. If a so-called baseline uses the same feature set, comparable model capacity, the same learning paradigm, or is expected to outperform the primary method, it must be promoted to a candidate primary or fallback method rather than hidden in the baseline set. Random search, simple simulation search, and handcrafted policy tuning can support baselines, warm starts, pilots, or tuning subroutines; they must not be the primary recommendation for a complex production problem unless the report gives explicit infeasibility evidence for higher-ceiling candidates. Any report containing results or benchmark comparisons must disclose every baseline separately: name, method logic, data/features used, tuned or fixed settings, tuning budget, deployability status, and its own metrics. Do not replace a set of baselines with a single aggregate baseline row or summary number. The main thread must study and respect the reviewer synthesis before presenting it: audit that the report's recommendation, branch score table, fallback, rigor check, benchmark taxonomy, and prose do not contradict each other, and revise the report before returning it if any inconsistency is found. Generate a static read-only HTML Method Selection Report comparing all suitable analytics, optimization, and training approaches, including pros, cons, data requirements, risks, performance ceiling, implementation difficulty, benchmark taxonomy, and a systematic recommendation. Return the HTML path/link plus a concise text choice prompt in chat. MUST ask the user to choose a method or raise objections, and MUST stop before deployment.
7. After the user selects or confirms a method, create an implementation plan using `references/deployment.md`. Before choosing a deployment scaffold, read `assets/asset_registry.md` and inspect every direct child folder under `assets/` that contains `asset_manifest.json`, including user-added custom assets. Choose the asset or asset combination that best matches the confirmed formulation, selected method, desired performance ceiling, implementation effort, dependencies, and industrial quality requirements. Built-in assets cover mathematical programming, reformulation, decomposition, stochastic/robust optimization, simulation optimization, CP/CP-SAT, DRL/MDP, and supervised learning, but they are examples rather than a closed list. Before generating the Deployment Confirmation Report, actively inspect the relevant deployment environment for the selected method: programming language/runtime versions, package availability and versions, solver availability, hardware/accelerator availability such as GPU when training or numerical acceleration matters, operating-system constraints, and license or install risks. For solver-based methods, perform a solver capability audit rather than a shallow import check: enumerate plausible Python environments, check Python APIs and command-line executables, test modeling-framework invocation routes such as Pyomo, CVXPY, PuLP, OR-Tools, or direct solver APIs, and run a tiny representative LP/MILP/CP/NLP solve to verify license, token, executable, and status handling. Distinguish `package importable`, `model object can be created`, `solver callable through the selected interface`, and `solver can complete a solve`. Prefer already installed, higher-performance solvers that actually pass a solve probe, especially commercial or specialized solvers such as Gurobi, CPLEX, Xpress, Mosek, SCIP, or HiGHS, when their license terms and the user's approval allow their use. If a solver fails with symptoms consistent with sandboxing, network restrictions, DNS resolution, token/license service access, license-server contact, executable path restrictions, or restricted filesystem access, rerun the minimal probe through the normal approval/escalation path before declaring the solver unavailable. For GPU-capable methods such as neural networks, RL, deep forecasting, large simulation, or differentiable optimization, explicitly probe GPU support and framework compatibility rather than assuming CPU execution; prefer available GPU acceleration when it is compatible and materially faster, and report the selected device. When conda, mamba, or micromamba is available, inspect plausible non-default Python environments before declaring dependencies missing; the user's default shell environment may not be the best environment. The environment survey must be method-specific; do not report irrelevant hardware or packages, but do report material missing capabilities. Missing packages, solvers, licenses, GPUs, GPU drivers, CUDA/MPS libraries, or accelerators must be reported as dependency actions, not used as silent reasons to ignore or reject a high-ceiling method. If the selected method requires missing capabilities, the Deployment Confirmation Report must ask whether the user approves installation/license setup or wants a faithful fallback. Run a pre-deployment reviewer check against the chosen formulation, method, implementation plan, selected asset contracts, and observed environment, then generate a static read-only HTML Deployment Confirmation Report describing the implementation approach and environment readiness. Return the HTML path/link plus a concise text approval prompt in chat. MUST stop and wait for user confirmation before writing or running deployment code. If the user requests changes, revise the implementation plan and repeat the static HTML report.
